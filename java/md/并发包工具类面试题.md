## CountDownLatch、CyclicBarrier、Semaphore、Exchanger

### 1. CountDownLatch、CyclicBarrier、Semaphore 的区别的使用场景？

**答案：**

| 工具类 | 作用 | 使用场景 | 是否可重用 | 关键方法 |
|--------|------|----------|------------|----------|
| **CountDownLatch** | 倒计时门闩，等待指定数量线程完成 | 主线程等待多个子线程完成 | ❌ 否 | `countDown()`、`await()` |
| **CyclicBarrier** | 循环栅栏，线程互相等待到达屏障 | 多线程分阶段任务，汇合后继续 | ✅ 是 | `await()` |
| **Semaphore** | 信号量，控制同时访问的线程数量 | 限流、资源池（如数据库连接池） | ✅ 是 | `acquire()`、`release()` |
| **Exchanger** | 交换器，两个线程交换数据 | 遗传算法、流水线等两两交换场景 | ✅ 是 | `exchange()` |

**原理对比：**

**CountDownLatch：**
```java
// 初始化计数器为3
CountDownLatch latch = new CountDownLatch(3);

// 每个子线程执行完调用 countDown()
latch.countDown(); // 计数器减1

// 主线程等待
latch.await(); // 计数器为0时才继续
```

**CyclicBarrier：**
```java
// 3个线程到达屏障后才继续
CyclicBarrier barrier = new CyclicBarrier(3);

// 每个线程调用 await()，到达后等待
barrier.await(); // 第3个到达后，所有线程继续

// 可以重置后重复使用
barrier.reset();
```

**Semaphore：**
```java
// 最多2个线程同时访问
Semaphore semaphore = new Semaphore(2);

semaphore.acquire();  // 获取许可（没有则阻塞）
semaphore.release();  // 释放许可
```

**CountDownLatch vs CyclicBarrier：**
- CountDownLatch：一个等多个（主线程等子线程），不可重用
- CyclicBarrier：多个互相等（子等子），可重用，支持回调

### 2. Exchanger 是什么？使用场景？

**答案：**

Exchanger 用于两个线程之间交换数据。

```java
Exchanger<String> exchanger = new Exchanger<>();

// 线程A
String dataA = "数据A";
String received = exchanger.exchange(dataA); // 阻塞等待线程B

// 线程B
String dataB = "数据B";
String received = exchanger.exchange(dataB); // 交换数据
```

**使用场景：**
- 遗传算法：父代交换基因
- 流水线：两个工作线程交换缓冲区
- 数据校对：两个线程交换数据后对比

**注意：**
- 只能用于两个线程
- 如果只有一个线程调用 exchange()，会一直阻塞

## Future vs CompletableFuture

### 3. Future 和 CompletableFuture 的区别？

**答案：**

| 特性 | Future (JDK 5) | CompletableFuture (JDK 8) |
|------|---------------|---------------------------|
| **获取结果** | `get()` 阻塞 | `get()` 阻塞，支持 `join()` |
| **手动完成** | ❌ 不支持 | ✅ `complete(T)` |
| **组合能力** | ❌ 不能组合 | ✅ `thenCombine`, `thenCompose` |
| **异常处理** | ❌ 需额外处理 | ✅ `exceptionally`, `handle` |
| **多任务等待** | ❌ 需自己实现 | ✅ `allOf()`, `anyOf()` |
| **异步执行** | ❌ 不支持 | ✅ `supplyAsync()`, `runAsync()` |

**Future 的局限：**
```java
Future<String> future = executor.submit(() -> "结果");

// 只能阻塞获取，无法链式处理
String result = future.get(); // 阻塞！

// 不能组合多个 Future
// 不能处理异常
// 不能主动完成
```

**CompletableFuture 的优势：**
```java
CompletableFuture<String> cf = CompletableFuture
    .supplyAsync(() -> fetchData())           // 异步获取数据
    .thenApply(data -> process(data))        // 链式处理
    .thenApply(result -> transform(result))   // 继续处理
    .exceptionally(ex -> "默认值");           // 异常处理

// 手动完成
cf.complete("提前完成");

// 组合两个任务
CompletableFuture<String> combined = cfA.thenCombine(cfB, (a, b) -> a + b);
```

### 4. CompletableFuture 的 allOf 和 anyOf 的实现原理？

**答案：**

**allOf 实现原理：**

等待所有 CompletableFuture 完成。

```java
public static CompletableFuture<Void> allOf(CompletableFuture<?>... cfs) {
    // 内部使用 CountDownLatch 或二叉树合并
    // 每个子任务完成后触发检查
    // 当所有任务完成时，返回的 CompletableFuture 完成
}
```

**实现机制：**
1. 创建一个结果 CompletableFuture<Void>
2. 为每个输入任务注册完成回调
3. 使用 AtomicInteger 计数未完成的任务数
4. 当计数器归零时，结果 CompletableFuture 完成

**anyOf 实现原理：**

返回最快的那个结果。

```java
public static CompletableFuture<Object> anyOf(CompletableFuture<?>... cfs) {
    // 内部使用 AtomicReference
    // 第一个完成的任务会设置结果
    // 后续完成的任务被忽略
}
```

**实现机制：**
1. 创建一个结果 CompletableFuture<Object>
2. 为每个输入任务注册完成回调
3. 使用 AtomicBoolean 标记是否已有任务完成
4. 第一个完成的任务设置结果，其他被忽略

**使用示例：**

```java
// allOf：等待所有完成
CompletableFuture<Void> all = CompletableFuture.allOf(task1, task2, task3);
all.join(); // 等待全部完成

// 获取所有结果
List<String> results = Stream.of(task1, task2, task3)
    .map(CompletableFuture::join)
    .collect(Collectors.toList());

// anyOf：取最快
CompletableFuture<Object> any = CompletableFuture.anyOf(task1, task2, task3);
Object result = any.get(); // 最快的那个结果
```

## ForkJoinPool 分治思想

### 5. ForkJoinPool 的分治思想是什么？工作窃取机制？

**答案：**

**分治思想（Divide and Conquer）：**

```
大任务
  ├── Fork: 拆分为子任务A
  │     └── 子任务A完成
  ├── Fork: 拆分为子任务B
  │     └── 子任务B完成
  └── Join: 合并结果
```

**核心组件：**

| 组件 | 说明 |
|------|------|
| `ForkJoinPool` | 线程池，管理工作线程 |
| `RecursiveTask<V>` | 有返回值的任务 |
| `RecursiveAction` | 无返回值的任务 |
| `fork()` | 异步执行子任务 |
| `join()` | 等待子任务完成并获取结果 |

**实现示例：**
```java
// 计算斐波那契或数组求和
class SumTask extends RecursiveTask<Long> {
    private long[] array;
    private int start, end;
    private static final int THRESHOLD = 10000;
    
    @Override
    protected Long compute() {
        // 任务足够小，直接计算
        if (end - start <= THRESHOLD) {
            long sum = 0;
            for (int i = start; i < end; i++) {
                sum += array[i];
            }
            return sum;
        }
        
        // 拆分任务
        int mid = (start + end) / 2;
        SumTask left = new SumTask(array, start, mid);
        SumTask right = new SumTask(array, mid, end);
        
        // 异步执行
        left.fork();
        right.fork();
        
        // 等待并合并
        return left.join() + right.join();
    }
}
```

**工作窃取（Work Stealing）：**

每个工作线程维护自己的双端队列（Deque）：
- **正常执行**：从队列头部取任务（LIFO）
- **任务做完后**：从其他线程队列尾部"窃取"任务（FIFO）

```
线程A队列: [任务1, 任务2, 任务3]
线程B队列: [任务4] (做完后空闲)

线程B从线程A队列尾部窃取: 窃取任务3
```

**优点：**
- 负载均衡：忙线程和闲线程之间自动均衡
- 减少竞争：窃取从队列尾部，避免与队列所有者冲突
- 提高效率：避免线程空闲等待

### 6. ForkJoinPool 的适用场景？

**答案：**

**适用场景：**
- ✅ **递归分治算法**：归并排序、快速排序、斐波那契
- ✅ **树形结构处理**：遍历、计算
- ✅ **大数据并行处理**：数组求和、最值查找
- ✅ **多维数据处理**：图像处理、矩阵运算

**不适用场景：**
- ❌ **IO密集型任务**：阻塞会影响整个线程池效率
- ❌ **任务间强依赖**：需要等待其他任务结果才能继续
- ❌ **数据量太小**：拆分开销 > 并行收益

### 7. RecursiveTask 和 RecursiveAction 的区别？

**答案：**

| 特性 | RecursiveTask<V> | RecursiveAction |
|------|------------------|-----------------|
| **返回值** | 有返回值 | 无返回值 |
| **使用场景** | 计算结果并返回（如求和、排序） | 仅执行操作（如打印、修改数组） |
| **方法** | `compute()` 返回 V | `compute()` 返回 void |

**RecursiveTask 示例：**
```java
class SumTask extends RecursiveTask<Long> {
    @Override
    protected Long compute() {
        // 计算并返回结果
        return left.join() + right.join();
    }
}
```

**RecursiveAction 示例：**
```java
class PrintTask extends RecursiveAction {
    @Override
    protected void compute() {
        // 执行操作，不返回
        invokeAll(leftTask, rightTask);
    }
}
```

## 线程池参数设计

### 8. 线程池的 corePoolSize、maximumPoolSize、workQueue 如何设计？

**答案：**

**核心参数设计：**

| 参数 | 设计原则 | 推荐值 |
|------|---------|--------|
| **corePoolSize** | 常驻线程数，根据任务类型 | CPU密集：N+1；IO密集：N*2 |
| **maximumPoolSize** | 最大线程数，防止资源耗尽 | core的2~4倍 |
| **workQueue** | 任务等待队列 | 有界队列（如ArrayBlockingQueue） |
| **keepAliveTime** | 非核心线程存活时间 | 60秒 |
| **handler** | 拒绝策略 | CallerRunsPolicy（推荐） |

**corePoolSize 计算：**

```java
// 1. CPU密集型（计算多，IO少）
// 公式：core = CPU核心数 + 1
// 原因：+1防止线程偶尔缺页或其他原因阻塞时，CPU空闲
int cpuCores = Runtime.getRuntime().availableProcessors();
int coreForCPU = cpuCores + 1;

// 2. IO密集型（IO多，计算少）
// 公式：core = CPU核心数 * 2
// 或：core = CPU核心数 / (1 - 阻塞系数)
// 原因：IO阻塞时CPU空闲，可增加线程提高利用率
int coreForIO = cpuCores * 2;

// 3. 混合型
// core = (IO时间 + CPU时间) / CPU时间 * CPU核心数
// 或拆分为两个线程池分别处理
```

**队列选择：**

| 队列 | 特点 | 推荐场景 |
|------|------|---------|
| **ArrayBlockingQueue** | 有界数组，需指定容量 | ✅ 生产环境推荐，防止OOM |
| **LinkedBlockingQueue** | 链表，默认无界 | ❌ 不推荐，容易堆积导致OOM |
| **SynchronousQueue** | 无容量，直接移交 | 任务执行快，线程池大的场景 |

**拒绝策略：**

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **AbortPolicy** | 抛出异常，快速失败 | 开发测试阶段 |
| **CallerRunsPolicy** | 由调用线程执行 | ✅ 生产推荐，削峰填谷 |
| **DiscardPolicy** | 静默丢弃 | ❌ 可能丢数据，不推荐 |
| **DiscardOldestPolicy** | 丢弃最老任务 | 新任务更重要时 |

### 9. 如何设计一个合理的线程池？

**答案：**

**推荐配置：**

```java
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    cpuCores + 1,                    // 核心线程数
    cpuCores * 2,                    // 最大线程数
    60L,                              // 空闲线程存活时间
    TimeUnit.SECONDS,
    new ArrayBlockingQueue<>(1000),   // 有界队列（容量1000）
    new ThreadFactory() {             // 自定义线程工厂
        private AtomicInteger count = new AtomicInteger(1);
        @Override
        public Thread newThread(Runnable r) {
            return new Thread(r, "mypool-" + count.getAndIncrement());
        }
    },
    new ThreadPoolExecutor.CallerRunsPolicy() // 拒绝策略
);
```

**设计步骤：**

1. **确定任务类型**：
   - CPU密集型：core = CPU + 1
   - IO密集型：core = CPU * 2

2. **确定队列大小**：
   - 根据内存和任务量设置队列大小
   - 使用有界队列防止OOM

3. **确定最大线程数**：
   - 通常是core的2~4倍
   - 根据系统资源调整

4. **选择拒绝策略**：
   - 推荐 CallerRunsPolicy，让调用者执行，自然降速

5. **添加监控**：
   - 监控活跃线程数、队列大小、任务完成数

6. **优雅关闭**：
   ```java
   executor.shutdown();
   if (!executor.awaitTermination(60, TimeUnit.SECONDS)) {
       executor.shutdownNow();
   }
   ```

**线程池类型选择：**

| 类型 | 使用场景 | 线程数 | 风险 |
|------|---------|--------|------|
| FixedThreadPool | 通用 | 固定 | 无界队列可能OOM |
| CachedThreadPool | 短任务 | 无限 | 线程数过多耗尽资源 |
| SingleThreadExecutor | 顺序执行 | 1 | 无界队列可能OOM |
| ScheduledThreadPool | 定时任务 | 固定 | - |

**⚠️ 阿里巴巴编码规范：**
- 禁止使用 Executors 的默认工厂方法创建线程池（无界队列风险）
- 必须使用 ThreadPoolExecutor 手动创建，指定有界队列

### 答题模板

**问：CountDownLatch、CyclicBarrier、Semaphore 有什么区别？**

**答：**

> 这三个都是 Java 并发包提供的同步工具类，但用途不同。
>
> **CountDownLatch** 是倒计时门闩，用于等待一组线程完成。比如主线程需要等待3个子线程加载完配置后才能启动，可以在主线程 `await()`，子线程完成后调用 `countDown()`。计数器归零后主线程继续。注意它是**一次性**的，计数器归零后不能重复使用。
>
> **CyclicBarrier** 是循环栅栏，用于让多个线程**互相等待**到达某个屏障点。比如多线程分阶段计算，每阶段完成后需要汇总结果再进入下一阶段。它支持**循环使用**（reset），还可以在到达屏障时执行回调操作。
>
> **Semaphore** 是信号量，用于**控制并发数量**。比如数据库连接池只有10个连接，可以用 Semaphore(10) 控制最多10个线程同时访问。`acquire()` 获取许可，`release()` 释放许可。公平模式还能防止饥饿。
>
> 简单说：CountDownLatch 是一个等多个，CyclicBarrier 是多个互相等，Semaphore 是控制能同时进入多少个。
