## 类加载过程

### 1. 类的生命周期是什么？类加载的完整过程？

**答案：**

类的生命周期包括7个阶段，其中**类加载（Loading）**包含5个子阶段：

```
加载(Loading) → 验证(Verification) → 准备(Preparation) → 解析(Resolution) → 初始化(Initialization) → 使用(Using) → 卸载(Unloading)
```

**类加载的5个阶段：**

| 阶段 | 工作内容 | 关键说明 |
|------|---------|---------|
| **加载** | 读取二进制字节流，在方法区创建类结构，堆中生成Class对象 | 可通过自定义类加载器控制加载过程 |
| **验证** | 文件格式验证、元数据验证、字节码验证、符号引用验证 | 确保字节码安全，不会危害JVM |
| **准备** | 为类变量（static）分配内存并设置默认初始值 | `static int a = 123;` 此阶段a=0，不是123 |
| **解析** | 将常量池中的符号引用替换为直接引用 | 类、接口、字段、方法的解析 |
| **初始化** | 执行类构造器`<clinit>()`方法，为类变量赋正确初始值 | 执行静态代码块和静态变量赋值 |

---

### 2. 加载阶段具体做什么？

**答案：**

**加载阶段的三件事：**

1. **通过一个类的全限定名获取定义此类的二进制字节流**
   - 来源：ZIP压缩包（JAR、EAR、WAR）、网络、运行时计算生成（动态代理）、其他文件生成（JSP）、数据库等

2. **将这个字节流所代表的静态存储结构转化为方法区的运行时数据结构**
   - 存储在方法区的类型信息

3. **在堆中生成一个代表这个类的`java.lang.Class`对象**，作为方法区这个类各种数据的访问入口

**Class对象的位置：**
- JDK 7及以前：方法区（永久代）
- JDK 8+：堆内存中（元数据在元空间，Class对象在堆）

---

### 3. 验证阶段包含哪些验证？为什么需要验证？

**答案：**

**验证阶段的4个步骤：**

| 验证类型 | 验证内容 | 目的 |
|---------|---------|------|
| **文件格式验证** | 魔数0xCAFEBABE、主次版本号、常量池合法性等 | 保证输入的字节流能被正确解析并存储于方法区 |
| **元数据验证** | 是否有父类、是否继承了final类、是否实现了接口方法 | 保证元数据信息符合Java语言规范 |
| **字节码验证** | 操作数栈和指令代码的匹配、类型转换有效性、跳转指令合法性 | 保证被校验类的方法在运行时不会危害JVM |
| **符号引用验证** | 通过全限定名能否找到对应的类、字段、方法是否存在、访问权限是否合法 | 为解析阶段做准备，确保解析能正常执行 |

**为什么需要验证：**
- Java语言本身是相对安全的语言（如数组边界检查）
- 但字节码未必由Java源码编译而来，可能由任何语言生成，甚至人工构造恶意字节码
- 验证阶段是JVM对自身保护的一项重要措施

### 4. 准备阶段和初始化阶段的区别？

**答案：**

**准备阶段：**
- 为类变量（static）分配内存并设置**默认初始值**（零值）
- 不包括实例变量，实例变量随对象实例化时分配在堆中

```java
public class Test {
    public static int a = 123;  // 准备阶段：a = 0（零值）
    public static final int b = 456;  // 准备阶段：b = 456（final常量直接赋值）
}
```

**初始化阶段：**
- 执行`<clinit>()`方法（类构造器）
- 按顺序执行静态变量赋值和静态代码块
- 是类加载过程的最后一步

```java
public class Test {
    public static int a = 123;  // 初始化阶段：执行赋值，a = 123
    
    static {  // 初始化阶段执行
        System.out.println("静态代码块");
        a = 456;
    }
}
```

### 5. 什么情况下会触发类的初始化？

**答案：**

**主动引用（必然触发初始化）：**

| 场景 | 示例 |
|------|------|
| **new实例化对象** | `new Test()` |
| **访问类的静态变量（非final）** | `Test.a`（final常量不会触发） |
| **调用类的静态方法** | `Test.method()` |
| **反射调用** | `Class.forName("Test")` |
| **初始化子类时，父类先初始化** | `class Sub extends Super`，new Sub()时Super先初始化 |
| **JVM启动时的主类** | `main()`方法所在的类 |
| **JDK 8默认方法实现类** | 接口定义了默认方法，实现类初始化 |

**被动引用（不会触发初始化）：**

```java
// 1. 通过子类引用父类的静态字段，不会导致子类初始化
System.out.println(SubClass.value);  // 只初始化父类

// 2. 通过数组定义来引用类，不会触发初始化
SuperClass[] sca = new SuperClass[10];  // 不会初始化SuperClass

// 3. 访问类的常量（final static）
System.out.println(ConstClass.HELLO);  // 编译期已放入常量池，不会初始化
```

## 双亲委派模型

### 6. 什么是双亲委派模型（Parent Delegation Model）？

**答案：**

**定义：**
当一个类加载器收到类加载请求时，它首先不会自己去尝试加载这个类，而是把这个请求委派给父类加载器去完成，每一层都是如此。只有当父加载器反馈自己无法完成这个加载请求（搜索范围内没有找到所需的类）时，子加载器才会尝试自己去加载。

**层次结构：**

```
        ┌─────────────────────────────────────────┐
        │           用户自定义类加载器              │
        │          (Application ClassLoader)      │
        │                   ▲                     │
        │                   │ 委派                │
        │                   │                     │
        │           ┌──────────────────┐          │
        │           │ 应用程序类加载器  │          │
        │           │ (App ClassLoader)│          │
        │           │      ▲           │          │
        │           │      │ 委派      │          │
        │           │      │           │          │
        │     ┌──────────────────────┐ │          │
        │     │   扩展类加载器         │ │          │
        │     │ (Extension ClassLoader)│          │
        │     │        ▲             │ │          │
        │     │        │ 委派        │ │          │
        │     │        │             │ │          │
        │  ┌───────────────────────────┐│          │
        │  │     启动类加载器            ││          │
        │  │  (Bootstrap ClassLoader)  ││          │
        │  └───────────────────────────┘│          │
        │        ↑ 最终尝试加载           │          │
        └─────────────────────────────────────────┘
```

**标准类加载器：**

| 类加载器 | 实现语言 | 作用范围 | 父加载器 |
|---------|---------|---------|---------|
| **Bootstrap ClassLoader** | C++（JVM内部） | `<JAVA_HOME>\lib`下的核心类库（rt.jar） | 无（顶层） |
| **Extension ClassLoader** | Java | `<JAVA_HOME>\lib\ext`或`java.ext.dirs`指定目录 | Bootstrap |
| **Application ClassLoader** | Java | 用户类路径（Classpath）上的类 | Extension |
| **Custom ClassLoader** | Java | 用户自定义加载路径 | Application |

### 7. 为什么要使用双亲委派模型？好处是什么？

**答案：**

**核心好处：保证Java核心库的类型安全，防止核心类被篡改。**

**具体优势：**

**1. 避免类的重复加载**
- 先让父加载器加载，如果父加载器已加载过，子加载器不需要重复加载
- 保证Java核心类在JVM中只有一份

**2. 保证Java核心API的安全性**
- 防止恶意代码替换核心类
- 例如：如果你写了一个`java.lang.String`类，由于双亲委派，加载请求会派给Bootstrap ClassLoader，它会加载JDK自带的String，而不是你写的恶意String

```java
// 如果没有双亲委派，用户编写了：
package java.lang;
public class String {
    // 恶意代码
}

// 如果App ClassLoader直接加载，会替换掉核心String类
// 有了双亲委派，Bootstrap ClassLoader优先加载JDK的String，保证安全
```

**3. 保证扩展性**
- 父加载器加载的类对子加载器可见
- 子加载器可以在此基础上进行扩展

**4. 代码复用**
- 基础的类由父加载器统一加载，子加载器共享使用

### 8. 双亲委派模型的源码实现？

**答案：**

**ClassLoader核心源码：**

```java
public abstract class ClassLoader {
    // 父类加载器
    private final ClassLoader parent;
    
    // 类加载的核心方法
    protected Class<?> loadClass(String name, boolean resolve) 
            throws ClassNotFoundException {
        // 1. 检查类是否已被加载
        Class<?> c = findLoadedClass(name);
        
        if (c == null) {
            // 2. 委派父类加载器加载
            if (parent != null) {
                c = parent.loadClass(name, false);
            } else {
                // 父加载器为null，使用Bootstrap ClassLoader
                c = findBootstrapClassOrNull(name);
            }
            
            // 3. 父加载器无法加载，自己尝试加载
            if (c == null) {
                c = findClass(name);  // 子类实现
            }
        }
        
        // 4. 链接（可选）
        if (resolve) {
            resolveClass(c);
        }
        
        return c;
    }
    
    // 子类需要实现的方法
    protected Class<?> findClass(String name) 
            throws ClassNotFoundException {
        throw new ClassNotFoundException(name);
    }
}
```

**关键流程：**
1. `findLoadedClass()` - 检查类是否已加载
2. `parent.loadClass()` - 委派父类加载器
3. `findBootstrapClassOrNull()` - 委派到Bootstrap
4. `findClass()` - 父类无法加载，自己加载（需子类实现）

### 9. 如何打破双亲委派模型？

**答案：**

**打破方式一：自定义类加载器重写`loadClass()`**
```java
public class BreakParentDelegation extends ClassLoader {
    
    @Override
    public Class<?> loadClass(String name) throws ClassNotFoundException {
        // 先自己尝试加载，加载不了再委派给父类
        Class<?> c = findLoadedClass(name);
        
        if (c == null) {
            // 自己的类自己加载
            if (name.startsWith("com.myapp")) {
                c = findClass(name);
            }
            
            // 其他类委派给父类
            if (c == null) {
                c = getParent().loadClass(name);
            }
        }
        
        return c;
    }
}
```

**打破方式二：线程上下文类加载器（Thread Context ClassLoader）**
- SPI（Service Provider Interface）机制打破了双亲委派
- JDBC、JNDI等使用SPI，父类加载器需要加载子类加载器的类

```java
// JDBC示例
Class.forName("com.mysql.jdbc.Driver");  // 由App ClassLoader加载

// 但Driver需要加载厂商实现的类，而Driver接口在JDK中（Bootstrap加载）
// 解决方案：使用Thread.currentThread().getContextClassLoader()
```

**打破方式三：OSGi模块化**
- 每个Bundle有自己的类加载器
- 形成网状结构，而非树形结构

**应用场景：**
- **Tomcat**：Web应用隔离，每个Web应用有自己的类加载器
- **Spring Boot DevTools**：重启时重新加载修改的类
- **OSGi**：模块热部署

### 10. Tomcat 为什么打破双亲委派？

**答案：**

**Tomcat的类加载结构：**

```
Common ClassLoader (共享类库)
    ↑
 Catalina ClassLoader (Tomcat本身)
    ↑
 Shared ClassLoader (Web应用共享)
    ↑
 WebApp ClassLoader (单个Web应用) ← 打破双亲委派
    ↑
    └── /WEB-INF/classes
    └── /WEB-INF/lib
```

**打破原因：**

**1. Web应用隔离**
- 不同Web应用可能使用同一个类的不同版本（如Spring 4 vs Spring 5）
- 如果遵循双亲委派，先加载的版本的类会被缓存，后加载的Web应用会报错

**2. 热部署（Hot Deployment）**
- Web应用修改后需要重新加载，而不重启Tomcat
- 如果遵循双亲委派，父加载器无法卸载已加载的类

**3. 优先加载Web应用自己的类**
- WebApp ClassLoader先尝试加载自己的类，加载不到再委派

**源码逻辑：**
```java
public class WebAppClassLoader extends URLClassLoader {
    
    @Override
    public Class<?> loadClass(String name) throws ClassNotFoundException {
        // 1. 先检查本地缓存
        Class<?> clazz = findLoadedClass0(name);
        
        // 2. 检查JVM缓存
        clazz = findLoadedClass(name);
        
        // 3. 检查是否系统类（避免核心类被篡改）
        if (clazz == null) {
            try {
                clazz = system.loadClass(name);
            } catch (ClassNotFoundException e) {
                // ignore
            }
        }
        
        // 4. 委派给父类（默认策略，可配置是否先本地）
        boolean delegateLoad = delegate || filter(name, true);
        if (delegateLoad) {
            clazz = parent.loadClass(name);
        }
        
        // 5. 本地加载（Web应用优先）
        if (clazz == null) {
            clazz = findClass(name);
        }
        
        // 6. 最后委派（如果没配置delegate）
        if (!delegateLoad && clazz == null) {
            clazz = parent.loadClass(name);
        }
        
        return clazz;
    }
}
```

## 自定义类加载器

### 11. 如何自定义类加载器？使用场景？

**答案：**

**自定义类加载器步骤：**

```java
public class MyClassLoader extends ClassLoader {
    
    private String classPath;
    
    public MyClassLoader(String classPath) {
        this.classPath = classPath;
    }
    
    @Override
    protected Class<?> findClass(String name) 
            throws ClassNotFoundException {
        // 1. 获取类的字节码
        byte[] classData = loadClassData(name);
        
        if (classData == null) {
            throw new ClassNotFoundException();
        }
        
        // 2. 调用defineClass将字节码转换为Class对象
        return defineClass(name, classData, 0, classData.length);
    }
    
    // 从文件系统加载类文件
    private byte[] loadClassData(String name) {
        String path = classPath + File.separatorChar 
            + name.replace('.', File.separatorChar) + ".class";
        
        try (InputStream is = new FileInputStream(path);
             ByteArrayOutputStream baos = new ByteArrayOutputStream()) {
            
            int bufferSize = 1024;
            byte[] buffer = new byte[bufferSize];
            int length;
            while ((length = is.read(buffer)) != -1) {
                baos.write(buffer, 0, length);
            }
            return baos.toByteArray();
            
        } catch (IOException e) {
            return null;
        }
    }
}
```

**使用场景：**

| 场景 | 说明 |
|------|------|
| **热部署/热替换** | 运行时替换类，不重启JVM |
| **代码加密保护** | 加载加密后的字节码，解密后加载 |
| **隔离加载** | 同一份字节码加载多份，实现隔离（如Tomcat） |
| **从不同源加载** | 网络、数据库、云端加载类 |
| **字节码增强** | 加载时修改字节码（AOP、插桩） |

### 答题模板

**问：什么是双亲委派模型？为什么要这样设计？**

**答：** 

> 双亲委派模型是Java类加载器的一种层次结构和工作方式。它要求除了顶层的Bootstrap ClassLoader外，每个类加载器在加载类时，首先将请求委派给父类加载器去完成，只有当父加载器无法完成加载请求时，子加载器才会尝试自己去加载。
>
> 这种设计有两大好处：
>
> **第一，保证Java核心类的安全性。** 比如我们自己写了一个`java.lang.String`类，如果没有双亲委派，Application ClassLoader可能会加载我们写的恶意String类，替换掉JDK的String。有了双亲委派，加载请求会先委派到Bootstrap ClassLoader，它会加载JDK自带的String类，保证核心类库不会被用户代码篡改。
>
> **第二，避免类的重复加载。** 父加载器加载过的类会被缓存，子加载器不需要重复加载，确保一个类在JVM中只存在一份。
>
> 不过这种模型也有局限性，所以在一些场景下需要打破它，比如Tomcat的Web应用隔离、SPI机制、OSGi热部署等。这些场景通过重写`loadClass()`方法或使用线程上下文类加载器来实现。
