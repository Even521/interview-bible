## CPU 100% 排查

### 1. 线上CPU 100%如何排查？完整步骤是什么？

**答案：**

**排查流程图：**
```bash
1. 确认问题：top 查看CPU高的进程ID
    ↓
2. 找到具体线程：top -Hp <pid> 查看占用CPU高的线程ID
    ↓
3. 线程ID转16进制：printf "%x\n" <tid>
    ↓
4. 查看堆栈：jstack <pid> | grep -A 100 <16进制tid>
    ↓
5. 定位问题代码
    ↓
6. 修复并验证
```

**详细步骤：**

**步骤1：找到CPU高的Java进程**
```bash
# 查看所有进程CPU使用情况
top
# 或
ps aux | grep java

# 输出示例：
PID USER  PR  NI VIRT  RES  SHR S %CPU %MEM TIME+ COMMAND
1234 root  20   0 4.5g  2.1g  15m R 99.9  25.0  2:30 java
```

**步骤2：找到具体占用CPU的线程**
```bash
# 查看进程1234的所有线程
top -Hp 1234

# 输出示例：
PID USER  PR  NI VIRT  RES  SHR S %CPU %MEM TIME+ COMMAND
5678 root  20   0 4.5g  2.1g  15m R 99.5  25.0  1:20 java
# 线程ID是5678，占用99.5%的CPU
```

**步骤3：线程ID转为16进制**
```bash
printf "%x\n" 5678
# 输出：162e
```

**步骤4：使用jstack查看线程堆栈**
```bash
# 打印进程堆栈，过滤出162e线程的堆栈（前后100行）
jstack 1234 | grep -A 100 "nid=0x162e"

# 或使用更完整的方式保存堆栈文件
jstack 1234 > thread_dump.txt
cat thread_dump.txt | grep -A 100 "nid=0x162e"
```

**步骤5：分析堆栈信息**
```java
"Thread-5" #29 prio=5 os_prio=0 tid=0x00007f8b2c162e nid=0x162e runnable [0x00007f8b1c9f5000]
   java.lang.Thread.State: RUNNABLE
        at com.example.service.CpuIntensiveTask.process(CpuIntensiveTask.java:45)
        at com.example.service.CpuIntensiveTask.run(CpuIntensiveTask.java:30)
        at java.lang.Thread.run(Thread.java:748)
```

**常见CPU 100%原因：**
| 原因 | 表现 | 解决方案 |
|------|------|---------|
| 死循环 | 代码逻辑错误导致无限循环 | 修复循环条件 |
| 正则表达式回溯 | 复杂正则导致灾难性回溯 | 优化正则，使用预编译 |
| 大量计算 | 复杂算法或数据处理 | 算法优化，分批处理 |
| Full GC频繁 | 垃圾回收线程占用CPU | 调优GC参数，排查内存泄漏 |
| 线程死锁 | 多个线程互相等待 | 解决死锁，优化锁粒度 |

## 内存泄漏排查

### 2. 线上内存泄漏如何排查？OOM如何分析？

**答案：**

**排查流程：**
```bash
1. 保留现场：内存溢出时自动生成dump文件
2. 获取dump：jmap -dump:format=b,file=heap.hprof <pid>
3. 分析dump：MAT工具分析 dominator tree、Histogram
4. 定位泄漏：查看GC Roots引用链，找到谁持有对象
5. 修复代码：及时释放引用，使用弱引用等
```

**JVM参数配置（OOM时自动生成dump）：**
```bash
# OOM时自动生成堆转储
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/data/logs/heapdump.hprof

# 发生OOM时执行脚本（可用于告警）
-XX:OnOutOfMemoryError="sh /data/scripts/alert.sh"
```

**内存泄漏常见场景：**

**场景1：静态集合持有对象**
```java
public class MemoryLeak {
    // 静态集合一直持有引用，不会被回收
    private static List<byte[]> cache = new ArrayList<>();
    
    public void add() {
        // 无限增长，永远不放
        cache.add(new byte[1024 * 1024]); // 1MB
    }
}
```

**场景2：未关闭资源**
```java
// 文件流、数据库连接、网络连接未关闭
public void readFile() throws IOException {
    InputStream is = new FileInputStream("file.txt");
    // 使用后未关闭，导致内存泄漏
    // 应该使用 try-with-resources
    try (InputStream is = new FileInputStream("file.txt")) {
        // 使用
    }
}
```

**场景3：ThreadLocal未清理**
```java
public void handle() {
    ThreadLocal<String> tl = new ThreadLocal<>();
    tl.set("data");
    // 使用完未remove，线程池场景下造成泄漏
    // 应该调用 tl.remove();
}
```

**场景4：内部类持有外部类引用**
```java
public class Outer {
    private byte[] data = new byte[1024 * 1024 * 100]; // 100MB
    
    public class Inner {
        // 隐含持有Outer.this引用
        public void print() {
            System.out.println(Outer.this.data.length);
        }
    }
    
    // 即使只new Inner()，Outer也不会被回收
    // 应该改为静态内部类
    public static class StaticInner {}
}
```

**场景5：缓存无大小限制**
```java
// Guava Cache或Caffeine配置不当
Cache<String, Object> cache = CacheBuilder.newBuilder()
    .maximumSize(1000)  // 需要设置大小限制
    .expireAfterWrite(10, TimeUnit.MINUTES)  // 设置过期时间
    .build();
```

**MAT分析步骤：**
```bash
1. Histogram：查看对象数量和占用内存，找出可疑类
2. Dominator Tree：查看支配树，找出占用内存最多的对象
3. Path to GC Roots：查看引用链，找到谁持有了对象
4. Leak Suspects：MAT自动分析的疑似泄漏点
```

## 常用排查工具

### 3. jstack、jmap、jstat、jcmd如何使用？

**答案：**

**jstack - 线程堆栈分析**
```bash
# 打印线程堆栈
jstack <pid>

# 打印堆栈到文件
jstack <pid> > thread_dump.txt

# 打印带锁信息的堆栈（JDK 8+）
jstack -l <pid>

# 检测死锁
jstack -l <pid> | grep -A 50 "deadlock"

# 示例输出：
"pool-1-thread-1" #15 prio=5 os_prio=0 tid=0x00007f4c2c002800 nid=0x3e47 waiting on condition [0x00007f4c1c9f4000]
   java.lang.Thread.State: WAITING (parking)
        at sun.misc.Unsafe.park(Native Method)
        - parking to wait for  <0x000000076b5c7d58> (a java.util.concurrent.locks.AbstractQueuedSynchronizer$ConditionObject)
```

**jmap - 内存分析**
```bash
# 查看堆内存概况
jmap -heap <pid>

# 生成堆转储文件（full gc前live）
jmap -dump:format=b,file=heap.hprof <pid>

# 只dump存活对象（推荐）
jmap -dump:live,format=b,file=heap.hprof <pid>

# 查看堆中对象统计
jmap -histo <pid> | head -20

# 示例输出（查看最多对象的类）：
 num     #instances      #bytes  class name
----------------------------------------------
   1:       2825962     452153920  [C
   2:        282595      67822800  java.lang.String
   3:         45823      52445376  [B
```

**jstat - GC监控**
```bash
# 每秒输出GC统计，持续10次
jstat -gc <pid> 1000 10

# 输出说明：
# S0C/S1C：Survivor区容量
# S0U/S1U：Survivor区已使用
# EC/EU：Eden区容量/已使用
# OC/OU：老年代容量/已使用
# MC/MU：Metaspace容量/已使用
# YGC/YGCT：Young GC次数/耗时
# FGC/FGCT：Full GC次数/耗时
# GCT：总GC耗时

# 查看编译统计
jstat -compiler <pid>

# 查看类加载统计
jstat -class <pid>
```

**jcmd - 综合诊断（JDK 8+推荐）**
```bash
# 列出所有Java进程
jcmd -l

# 查看进程详情
jcmd <pid> VM.version
jcmd <pid> VM.flags  # 查看JVM参数
jcmd <pid> GC.run     # 手动触发GC

# 生成堆转储
jcmd <pid> GC.heap_dump /data/heap.hprof

# 生成线程dump
jcmd <pid> Thread.print

# 查看性能统计
jcmd <pid> PerfCounter.print
```

**jinfo - 查看JVM参数**
```bash
# 查看所有JVM参数
jinfo <pid>

# 查看特定参数
jinfo -flag MaxHeapSize <pid>

# 动态修改参数（部分可修改）
jinfo -flag +/-PrintGCDetails <pid>
```

## Arthas 实战

### 4. Arthas如何使用？相比传统工具有什么优势？

**答案：**

**Arthas简介：**
阿里巴巴开源的Java诊断工具，无需重启、无需修改代码，实时诊断线上问题。

**安装启动：**
```bash
# 下载
curl -O https://arthas.aliyun.com/arthas-boot.jar

# 启动（自动识别Java进程）
java -jar arthas-boot.jar

# 或者指定进程ID
java -jar arthas-boot.jar <pid>
```

**常用命令：**

**dashboard - 系统仪表盘**
```bash
$ dashboard
ID   NAME                      GROUP      PRIORITY STATE   %CPU  TIME     INTERRUPTED DAEMON
122  pool-1-thread-1           main       5        RUNNABLE 18    0:1:20   false       false
1    main                      main       5        TIMED_WAI 87    12:20:30 false       false

# 显示线程、JVM、GC、内存使用情况
# 按Ctrl+C退出
```

**thread - 线程诊断（CPU问题排查）**
```bash
# 查看所有线程
$ thread

# 查看CPU占用最高的前3个线程
$ thread -n 3

# 查看线程ID为122的堆栈
$ thread 122

# 查看死锁线程
$ thread -b

# 示例：直接定位CPU高的线程
$ thread -n 1
"pool-1-thread-5" Id=122 cpuUsage=52% RUNNABLE
    at com.example.service.CpuIntensiveTask.process(CpuIntensiveTask.java:45)
```

**jad - 反编译代码**
```bash
# 反编译类（查看线上运行代码）
$ jad com.example.service.UserService

# 反编译方法
$ jad com.example.service.UserService getUserById

# 查看类的类加载器
$ jad -c com.example.service.UserService
```

**watch - 监控方法**
```bash
# 监控方法入参和返回值
$ watch com.example.service.UserService getUserById '{params,returnObj}' -x 2

# 监控方法执行时间超过100ms的
$ watch com.example.service.UserService * '#cost>100' -x 2

# 监控异常
$ watch com.example.service.UserService * '{params[0],throwExp}' -e

# 条件监控（只监控userId=100的）
$ watch com.example.service.UserService getUserById '{params,returnObj}' 'params[0]==100' -x 2
```

**trace - 方法内部调用路径**
```bash
# 跟踪方法调用链和耗时
$ trace com.example.service.UserService getUserById '#cost>10'

# 示例输出
`---[10.1234ms] com.example.service.UserService:getUserById()
    +---[2.1111ms] com.example.dao.UserDao:selectById()
    `---[5.2222ms] com.example.cache.RedisCache:get()
```

**stack - 调用栈**
```bash
# 查看方法被谁调用了
$ stack com.example.service.UserService getUserById

# 查看调用者分布
$ stack com.example.service.UserService getUserById -n 5
```

**tt - 时空隧道（记录并重放）**
```bash
# 记录方法调用
$ tt -t com.example.service.UserService getUserById

# 查看记录列表
$ tt -l

# 根据索引重新执行
$ tt -i 1006 -p

# 回放并修改参数
$ tt -i 1006 -p --params '#params[0]=99999'
```

**sc - 查看类信息**
```bash
# 查看加载的类
$ sc com.example.service.*

# 查看类详细信息
$ sc -d com.example.service.UserService

# 查看类从哪个jar加载
$ sc -c com.example.service.UserService
```

**sm - 查看方法信息**
```bash
# 查看类的方法
$ sm com.example.service.UserService

# 查看方法详细信息
$ sm -d com.example.service.UserService getUserById
```

**ognl - 执行表达式**
```bash
# 获取静态字段值
$ ognl '@com.example.config.AppConfig@staticField'

# 调用静态方法
$ ognl '@com.example.util.DateUtil@getCurrentTime()'

# 修改静态字段（危险，谨慎使用）
$ ognl '@com.example.config.AppConfig@switch=false'
```

**vmtool - 强制GC**
```bash
# 强制GC
$ vmtool --action forceGc

# 获取对象实例
$ vmtool --action getInstances --className com.example.model.User
```

**Arthas优势：**

| 特性 | 传统工具(jstack/jmap) | Arthas |
|------|---------------------|--------|
| **侵入性** | 需要生成dump文件，暂停应用 | 无侵入，应用无感知 |
| **实时性** | 静态快照，无法连续监控 | 实时查看，动态更新 |
| **操作难度** | 命令复杂，需要导出文件分析 | 交互式命令，结果直接展示 |
| **功能丰富** | 仅堆栈、内存 | 反编译、方法监控、热更新等 |
| **环境要求** | 需要服务器权限下载dump | 只需要JVM attach权限 |

## 面试题

### 5. 线上服务突然CPU飙升，你的排查思路是什么？

**答案：**

> **快速排查步骤：**
> 
> **第一步：确认问题范围**，通过监控告警查看是否单台机器还是全部机器，确定是代码问题还是流量问题。
> 
> **第二步：定位具体进程**，使用`top`命令查看哪个Java进程占用CPU高，记录进程ID。
> 
> **第三步：找到具体线程**，使用`top -Hp <pid>`查看该进程内哪个线程占用CPU最高，记录线程ID。
> 
> **第四步：线程ID转16进制**，使用`printf "%x\n" <tid>`将线程ID转为16进制。
> 
> **第五步：查看线程堆栈**，使用`jstack <pid> | grep -A 100 <16进制tid>`查看该线程正在执行的代码，定位到具体方法。
> 
> **第六步：分析代码**，查看源码是否存在死循环、正则回溯、复杂计算等问题。
> 
> **如果使用Arthas**：直接用`thread -n 3`就能看到CPU最高的3个线程及其堆栈，一步到位。
> 
> **如果是Full GC导致**：使用`jstat -gc <pid> 1000`查看GC频率，如果是GC问题需要进一步用`jmap`分析内存。

### 6. 如何分析内存泄漏？写出具体命令和工具使用步骤。

**答案：**

> **内存泄漏排查步骤：**
> 
> **1. 保留现场**
> 配置JVM参数让OOM时自动生成dump：
> `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/data/logs/`
> 
> **2. 获取dump文件**
> 如果进程还在运行： `jmap -dump:live,format=b,file=heap.hprof <pid>`
> 或： `jcmd <pid> GC.heap_dump /data/heap.hprof`
> 
> **3. 分析dump文件**
> 使用MAT（Memory Analyzer Tool）或VisualVM打开hprof文件：
> - **Histogram**：按类统计实例数量和内存占用，找出占用内存最多的类
> - **Dominator Tree**：查看支配树，找出大对象的引用链
> - **Leak Suspects**：MAT自动分析的疑似内存泄漏报告
> - **Path to GC Roots**：查看对象的GC Roots引用链，找到谁持有了这个对象
> 
> **4. 定位泄漏点**
> 重点关注：
> - 静态集合（HashMap、List）一直put不remove
> - ThreadLocal使用后不remove
> - 监听器或回调注册后不注销
> - 数据库连接、IO流未关闭
> - 缓存无大小限制一直增长
> 
> **5. 修复验证**
> 修复代码后重新发布，通过监控观察内存使用曲线是否恢复正常。

### 7. Arthas 和 jstack/jmap 有什么区别？线上诊断推荐使用哪个？

**答案：**

> **主要区别：**
> 
> **传统工具（jstack/jmap）：**
> - jstack生成线程dump文件，jmap生成堆dump文件，都是静态快照
> - 需要下载文件到本地用MAT等工具分析
> - jmap执行时会导致Full GC（生产环境慎用`-dump:live`）
> - 操作简单但需要文件传输、多工具配合
> 
> **Arthas：**
> - 实时诊断，无需生成文件，直接查看运行时状态
> - 功能丰富：dashboard系统面板、thread定位CPU问题、watch方法监控、jad反编译、tt时空隧道等
> - 无侵入，对应用性能影响小
> - 命令交互式，结果直接展示在终端
> 
> **推荐：**
> - **线上问题快速定位推荐Arthas**，特别是CPU100%、方法耗时、热更新等场景
> - **内存泄漏分析还是需要jmap生成dump后用MAT分析**，因为Arthas查看大对象不如MAT直观
> - 两者结合使用：先用Arthas快速定位问题范围，必要时再用jmap生成dump详细分析
