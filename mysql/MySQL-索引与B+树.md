## 索引概述

### 什么是索引？

索引是数据库中用于**加速数据查询**的数据结构，相当于书籍的目录。

```sql
-- 创建索引
CREATE INDEX idx_name ON users(name);

-- 创建复合索引
CREATE INDEX idx_name_age ON users(name, age);
```

### 常见的索引数据结构

| 数据结构 | 特点 | MySQL支持 |
|---------|------|----------|
| **B+ Tree** | 平衡多路搜索树，适合范围查询 | **默认，主流** |
| Hash | O(1) 查找，不支持范围查询 | Memory引擎 |
| B-Tree | 所有节点存储数据，不稳定 | 早期使用 |
| Red-Black Tree | 二叉树，树高过高 | 未使用 |
| Skip List | 跳表，适合内存数据库 | Redis使用 |

## 为什么选择 B+ 树

### MySQL 选择 B+ 树的 6 大原因

#### 1. 磁盘 I/O 优化

**核心问题**：数据库数据存储在磁盘，磁盘 I/O 是瓶颈。

```
B+ 树特点：每一层节点按页(Page)存储
- InnoDB 页大小：16KB
- 每个节点可以存储更多关键字
- 树高度更低，减少磁盘读取次数

B+ 树 vs 红黑树（假设存储1000万数据）：
┌─────────────────────────────────────────┐
│ 红黑树（二叉树）                          │
│   高度：~24层                            │
│   最坏需要24次磁盘I/O                     │
│                                         │
│ B+ 树（假设每个节点100个关键字）           │
│   高度：3-4层                            │
│   最多需要4次磁盘I/O                      │
└─────────────────────────────────────────┘
```

#### 2. 查询稳定性

**B+ 树所有叶子节点在同一层**，保证任何数据查找的磁盘 I/O 次数相同。

```java
// 查找稳定性对比

// 红黑树：最坏 O(log n)，但树高不稳定
// B+ 树：固定树高（通常是3-4层），查询性能稳定
```

#### 3. 范围查询优化

**B+ 树的叶子节点形成有序链表**，范围查询只需遍历叶子节点。

```
B+ 树范围查询 [20, 50]：

     [30]          ← 根节点（索引层）
    /   \
 [10,20] [40,50]   ← 非叶子节点（索引层）
  ↓   ↓   ↓   ↓
[10]-[20]-[30]-[40]-[50]  ← 叶子节点（数据层）
                      ↑
                找到20，向右遍历到50即完成
                时间复杂度：O(log n + m)

Hash索引范围查询：
- 不支持！必须逐条查询每个值
```

#### 4. 空间局部性

**非叶子节点不存储数据，只存储索引关键字**，一页能容纳更多索引。

```
┌─────────────────────────────────────────┐
│ B-Tree 节点结构（每个节点存储数据）        │
│ ┌─────┬───────┬─────┬───────┬─────┐   │
│ │ Key │ Data  │ Key │ Data  │ Key │   │
│ └─────┴───────┴─────┴───────┴─────┘   │
│ 数据占用空间，每页存储索引数量有限          │
└─────────────────────────────────────────┘

┌─────────────────────────────────────────┐
│ B+ Tree 节点结构（非叶子节点只存索引）      │
│ ┌─────┬─────┬─────┬─────┬─────┬─────┐ │
│ │ Key │ Key │ Key │ Key │ Key │ Key │ │
│ └─────┴─────┴─────┴─────┴─────┴─────┘ │
│ 数据只存在叶子节点，非叶子节点更紧凑         │
│ 扇出更大，树高更低                        │
└─────────────────────────────────────────┘
```

#### 5. 全表扫描优化

**叶子节点链表结构**使全表扫描只需顺序读取，充分利用磁盘预读。

```sql
-- 全表扫描（SELECT * FROM users）
-- B+树：顺序扫描叶子节点链表，磁头移动最小
-- Hash表：跳跃式访问，随机I/O
```

#### 6. 插入删除高效

**B+ 树采用分裂合并策略**，保持树的平衡。

```
插入操作：

B+树：
- 找到叶子节点插入
- 节点满则分裂
- 向上传播分裂
- 时间复杂度：O(log n)

红黑树：
- 插入后需要旋转和变色
- 维护平衡成本更高
```

### 数据结构对比总结

| 特性 | B+ Tree | B-Tree | Hash | Red-Black Tree |
|------|---------|--------|------|----------------|
| **查找** | O(log n) | O(log n) | O(1) | O(log n) |
| **范围查询** | ✅ 高效 | ⚠️ 一般 | ❌ 不支持 | ❌ 困难 |
| **磁盘友好** | ✅ 是 | ✅ 是 | ❌ 否 | ❌ 否 |
| **树高** | 低（3-4层） | 较高 | - | 高（~log n）|
| **稳定性** | 稳定 | 稳定 | 不稳定 | 稳定 |
| **顺序访问** | ✅ 高效 | ⚠️ 需中序遍历 | ❌ 不支持 | ❌ 困难 |

## B+ 树结构详解

### 节点结构

```
B+ 树（阶数为3，即 m=3）

                    [50, 100]                    ← 根节点（Internal Node）
                   /     |     \
            [20, 30]  [70, 80]  [120, 150]       ← 内部节点（Internal Node）
            /  |  \    / | \    /   |   \
          [10] [25] [40] ┆  [90] [110] [130] [160] ← 叶子节点（Leaf Node）
           ↓    ↓    ↓       ↓     ↓     ↓     ↓
          数据   数据  数据    数据   数据   数据   数据

特点：
1. 非叶子节点只存储索引键值，叶子节点存储实际数据
2. 叶子节点之间用双向链表连接
3. 所有叶子节点在同一层（平衡）
```

### InnoDB B+ 树实现

```sql
-- 查看 InnoDB 页大小
SHOW VARIABLES LIKE 'innodb_page_size';
-- 默认 16384 bytes = 16KB

-- 一页可以存储多少索引？
-- 假设主键是 BIGINT(8字节) + 指针(6字节) = 14字节
-- 索引数量 = 16384 / 14 ≈ 1170 个

-- 一层 B+ 树可以存储多少数据？
-- 第一层（根节点）：1170 个指针
-- 第二层：1170 * 1170 = 1,368,900 个指针
-- 第三层（叶子节点）：假设每页存16条数据记录
-- 总数据量 = 1170 * 1170 * 16 ≈ 2200万条
```

### 聚集索引 vs 非聚集索引

#### 聚集索引（Clustered Index）

```sql
-- 主键索引就是聚集索引
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    age INT
) ENGINE=InnoDB;

-- B+ 树结构：
-- 叶子节点存储完整的行数据
-- 数据行按主键顺序物理存储

示例查询：
SELECT * FROM users WHERE id = 5;

执行过程：
1. 从根节点开始查找
2. 定位到叶子节点
3. 从叶子节点读取完整行数据
4. 无需回表，效率高
```

#### 非聚集索引（Secondary Index）

```sql
-- 创建普通索引
CREATE INDEX idx_name ON users(name);

-- B+ 树结构：
-- 叶子节点存储的是：索引键 + 主键值

示例查询：
SELECT * FROM users WHERE name = '张三';

执行过程：
1. 在 idx_name 索引中查找 '张三'
2. 找到对应的主键值（如 id=10）
3. **回表**：根据主键值去主键索引查找完整数据
4. 返回结果
```

### 回表现象

```sql
-- 回表示例
SELECT * FROM users WHERE name = '张三';
-- 1. 查二级索引 idx_name 得到 id=10
-- 2. 再根据 id=10 查主键索引得到完整数据

-- 避免回表（覆盖索引）
SELECT name, age FROM users WHERE name = '张三';
-- 如果 idx_name 是复合索引 (name, age)
-- 那么叶子节点包含 (name, age, id)
-- 无需回表，直接从索引返回数据
```

### 联合索引的 B+ 树结构

```sql
CREATE INDEX idx_name_age ON users(name, age);

-- B+ 树按 (name, age) 排序

         [('A',20)]        [('C',25)]
        /           \      /           \
[('A',18),('A',20)] [('B',22)] [('C',25),('D',30)]

-- 最左前缀法则：
-- WHERE name = 'A' AND age = 20  ✅ 使用索引
-- WHERE name = 'A'                ✅ 使用索引
-- WHERE age = 20                  ❌ 无法使用索引（跳过name）
-- WHERE name = 'A' OR age = 20    ❌ 无法使用索引
```

## 索引类型

### 1. 主键索引（Primary Key）

```sql
-- 自动创建 B+ 树聚集索引
CREATE TABLE users (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50)
);
-- 特点：唯一、非空、只能有一个
```

### 2. 唯一索引（Unique）

```sql
CREATE UNIQUE INDEX idx_email ON users(email);
-- 特点：唯一约束，允许NULL，可以有多个
```

### 3. 普通索引（Normal）

```sql
CREATE INDEX idx_name ON users(name);
-- 特点：没有唯一约束，提高查询速度
```

### 4. 全文索引（Full Text）

```sql
CREATE FULLTEXT INDEX idx_content ON articles(content);
-- 特点：支持文本搜索，类似搜索引擎
-- 使用：WHERE MATCH(content) AGAINST('keyword')
```

### 5. 空间索引（Spatial）

```sql
CREATE SPATIAL INDEX idx_location ON shops(location);
-- 特点：支持地理数据查询（GIS）
```

## 索引优化实践

### 1. 索引设计原则

```sql
✅ 应该创建索引的场景：
- 主键、外键
- WHERE 条件字段
- ORDER BY、GROUP BY 字段
- 连接表JOIN的字段
- DISTINCT字段（去重后计数）

❌ 不应该创建索引的场景：
- 数据量小（<1000条）
- 经常更新的字段（如计数器）
- 区分度低的字段（如性别）
- 不在查询中使用的字段
- TEXT/BLOB大字段（可建前缀索引）
```

### 2. 索引优化案例分析

```sql
-- 案例1：最左前缀优化
-- 联合索引 (a, b, c)
CREATE INDEX idx_abc ON table(a, b, c);

WHERE a = 1 AND b = 2 AND c = 3  -- ✅ 完全使用
WHERE a = 1 AND b = 2            -- ✅ 使用 a,b
WHERE a = 1                       -- ✅ 使用 a
WHERE b = 2 AND c = 3            -- ❌ 不使用（缺少a）
WHERE a = 1 AND c = 3            -- ⚠️ 只使用 a

-- 案例2：范围查询导致索引失效
CREATE INDEX idx_age ON users(age, name);

WHERE age > 20 AND name = '张三'  
-- 只能使用 age 索引，name 列失效
-- 优化：调整索引顺序 (name, age) 或拆分条件

-- 案例3：函数导致索引失效
WHERE DATE(create_time) = '2024-01-01'
-- ❌ 索引失效，因为对字段使用了函数

WHERE create_time >= '2024-01-01 00:00:00' 
  AND create_time < '2024-01-02 00:00:00'
-- ✅ 改写后可以使用索引

-- 案例4：隐式类型转换
CREATE INDEX idx_phone ON users(phone); -- phone是varchar

WHERE phone = 13800138000
-- ❌ 整数与字符串比较，发生隐式转换，索引失效

WHERE phone = '13800138000'
-- ✅ 类型一致，使用索引
```

### 3. 索引覆盖

```sql
-- 覆盖索引：查询的列都在索引中，无需回表

-- 创建复合索引
CREATE INDEX idx_name_age ON users(name, age);

-- 查询列被索引覆盖
EXPLAIN SELECT name, age FROM users WHERE name = '张三';
-- Extra: Using index 表示使用了覆盖索引

-- 对比非覆盖查询
EXPLAIN SELECT * FROM users WHERE name = '张三';
-- Extra: NULL 表示需要回表查询
```

### 4. 索引下推（Index Condition Pushdown）

```sql
-- MySQL 5.6+ 优化特性
-- 在存储引擎层过滤数据，减少回表次数

CREATE INDEX idx_name_age ON users(name, age);

-- 查询
SELECT * FROM users WHERE name LIKE '张%' AND age = 20;

-- 不使用索引下推：
-- 1. 找到所有姓张的记录（name LIKE '张%'）
-- 2. 全部回表查询age
-- 3. 过滤age=20的

-- 使用索引下推：
-- 1. 找到姓张的记录
-- 2. 在索引中直接判断age是否等于20
-- 3. 只回表符合条件的记录
```

## 常见问题

### Q1：为什么 MyISAM 也使用 B+ 树，但 InnoDB 性能更好？

```
MyISAM vs InnoDB 索引结构：

┌─────────────────────────────────────────────────┐
│ MyISAM                                          │
│ - 数据和索引分离存储                             │
│ - B+ 树叶子节点存储的是数据文件地址                │
│ - 查找到地址后还需要去数据文件读取                  │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│ InnoDB                                          │
│ - 索引即数据（聚簇索引）                          │
│ - B+ 树叶子节点直接存储完整行数据                  │
│ - 主键查询性能极高                                │
│ - 二级索引查找到主键后需要回表                      │
└─────────────────────────────────────────────────┘

结论：
- InnoDB 聚簇索引减少了一次磁盘I/O
- InnoDB 支持事务、行级锁、MVCC
```

### Q2：哈希索引查找更快，为什么MySQL默认用B+树？

```
哈希索引的缺陷：

1. ❌ 无法范围查询
   WHERE age > 20 AND age < 30
   哈希索引必须逐个计算哈希值
   B+ 树直接在范围内遍历

2. ❌ 不支持ORDER BY
   哈希是无序的，无法排序

3. ❌ 不支持联合索引的最左前缀
   哈希是对整个联合索引计算哈希值
   不能只查某一列

4. ❌ 存在哈希冲突
   需要额外的冲突处理

适用场景：
- 等值查询且不需要排序
- Memory引擎的临时表
```

### Q3：主键为什么要用自增整数？

```
自增整数主键优势：

1. ✅ 存储空间小
   INT: 4字节  vs BIGINT: 8字节  vs UUID: 36字节

2. ✅ 插入性能高
   自增ID顺序插入，避免页分裂
   UUID随机插入，频繁页分裂，碎片多

3. ✅ 查询效率高
   整数比较比字符串比较快
   整数索引的B+树更紧凑

4. ✅ B+树结构稳定
   自增ID保证永远在后面插入，树更平衡
   
插入顺序影响：
自增ID： [1],[2],[3],[4],[5],[6]  -- 顺序插入，页填满后新建页
UUID：   [随机],[随机],[随机]      -- 随机插入，可能导致页分裂
```

### Q4：什么是索引失效？如何避免？

```sql
常见索引失效场景：

1. 字段上使用函数或表达式
   WHERE YEAR(create_time) = 2024 -- 失效
   WHERE create_time >= '2024-01-01' -- 有效

2. 隐式类型转换
   WHERE phone = 13800138000 -- phone是varchar，失效

3. 使用 !=、<>、NOT IN（可能失效）
   WHERE age != 20 -- 可能失效

4. LIKE 以%开头
   WHERE name LIKE '%张%' -- 失效
   WHERE name LIKE '张%' -- 可以使用索引

5. OR 条件使用不当
   WHERE name = '张三' OR age = 20
   -- name和age都有索引可能用，否则失效

6. 联合索引不满足最左前缀

7. 数据量小，优化器认为全表扫描更快

查看执行计划：
EXPLAIN SELECT * FROM users WHERE name = '张三';
-- 关注：type（访问类型）、key（使用的索引）、rows（扫描行数）、Extra
```

### Q5：InnoDB一棵B+树可以存放多少行数据？

```sql
-- 计算示例（假设）：
-- 主键BIGINT：8字节
-- 指针：6字节
-- InnoDB页大小：16KB

-- 非叶子节点每页可存储索引数：
n = 16384 / (8 + 6) ≈ 1170

-- 假设B+树高度为3：
-- 第一层（根节点）：1170个指针
-- 第二层：1170个节点，共1170×1170=1,368,900个指针
-- 第三层（叶子节点）：假设每行数据1KB（包含索引和其他数据）
-- 叶子节点每页可存：16384 / 1024 = 16 行

-- 总数据量：
-- 1170 × 1170 × 16 ≈ 21,902,400 行（约2200万）

实际生产中：
- B+树通常维持在3-4层（树高可控）
- 单表超过2000万建议分表或分区
```

## 总结

### B+ 树核心优势记忆口诀

> **"树矮枝多存数据，叶链表连好顺序，聚焦索引省IO"**

1. **树矮**：多路平衡，树高固定（3-4层）
2. **枝多**：非叶子节点不存数据，存放更多索引
3. **叶链表**：叶子节点双向链表连接，范围查询高效
4. **聚焦索引**：InnoDB数据直接存叶子节点，查询快
5. **省IO**：一次磁盘读取一页，减少I/O次数

### 面试金句

> "MySQL选择B+树作为索引结构，主要因为B+树的矮胖结构可以减少磁盘I/O次数，叶子节点的双向链表支持高效的范围查询，且聚集索引设计使得主键查询性能最优。相比二叉树，B+树更适合磁盘存储；相比B树，B+树所有数据在叶子节点且形成有序链表，更适合数据库的范围扫描和全表遍历场景。"
